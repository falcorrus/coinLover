# Technical Guide: CoinLover

## 🛠 Architecture Overview
Текущая архитектура проекта представляет собой гибридное решение: современный Frontend, который общается с базой данных (Google Sheets) не напрямую, а через собственный проксирующий Backend. Вся система физически работает на VPS-сервере (`server.reloto.ru`).

### 1. Frontend (Пользовательский интерфейс)
Это то, что загружается в браузере пользователя, когда он открывает `coinlover.ru` или `coin.reloto.ru`.
- **Технологии:** React 19, TypeScript, Vite (сборщик).
- **Стилизация:** Tailwind CSS v4.
- **Ключевые библиотеки:** `@dnd-kit/core` (для Drag-and-Drop механики монет), `framer-motion` (для анимаций), `lucide-react` (иконки).
- **Расположение:**
  - *Исходный код:* Папка `src/` в корне проекта.
  - *На сервере:* Внутри Docker-контейнера `coinlover` (или `dev-coinlover`). При деплое Vite собирает статические файлы (HTML/JS/CSS) в папку `dist/`.

### 2. Backend (VPS Proxy Server)
Это "мозг", связывающий фронтенд и таблицы. Отказ от Google Apps Script и Python-бэкенда в пользу легковесного Node.js сервера был сделан для обхода CORS и повышения безопасности.
- **Технологии:** Node.js, Express (микро-фреймворк), `googleapis` (библиотека Google для работы с API).
- **Главный файл:** `api/sheets.ts`. Это единая точка входа (`/api/sheets`), обрабатывающая GET (загрузка данных) и POST (сохранение транзакций, обновление настроек, инициализация таблиц).
- **Обертка:** Файл `server.ts`. Поднимает Express-сервер на 80 порту внутри контейнера, раздает статику из папки `dist/` и перенаправляет все запросы `/api/*` в `api/sheets.ts`.
- **Расположение:**
  - *Исходный код:* Папка `api/` и файл `server.ts`.
  - *На сервере:* Запускается внутри Docker-контейнера с помощью утилиты `tsx`.

### 3. Database (База данных)
Вместо традиционного SQL используются Google Таблицы.
- **Роли таблиц:**
  - **Мастер-таблица:** (ID: `1IQCs35RQlMMQsGB-CRczJeuRqa8WIxW4Sy_kjZyHP2M`). Вкладка `Users` хранит список пользователей, их ID таблиц и даты окончания подписки (для проверки прав доступа на бэкенде). Сам вход в приложение полностью децентрализован: клиент вводит прямую ссылку на свою таблицу, из которой парсится `ssId`, а бэкенд на сервере лишь валидирует доступ и сверяет лимиты.
  - **Пользовательские таблицы:** Создаются пользователями. Содержат листы `Configs` (кошельки, категории, настройки) и `Transactions` (история операций).
- **Авторизация (Service Account):** Происходит сервер-сервер. Бэкенд на VPS стучится в Google API от имени Сервисного Аккаунта (`coinlover-service-acc@baonlineru.iam.gserviceaccount.com`). Пользователи дают доступ этому email'у, а ключи лежат в скрытом файле `.env` на VPS.

### 4. Infrastructure & Deployment (Инфраструктура)
То, как код попадает на сервер и работает там.
- **Технологии:** Docker, Bash, Nginx, Git.
- **Скрипт `deploy.sh`:** Выполняет следующие шаги:
  1. Проверяет TypeScript-типы и собирает фронтенд (`vite build`).
  2. Пушит код в GitHub (в ветку `preview` или `main`).
  3. Подключается по SSH к `server.reloto.ru`.
  4. Забирает свежий код из GitHub.
  5. Генерирует `docker-compose.yml` (пробрасывая секреты из `.env` через директиву `env_file`).
  6. Собирает Docker-образ (на основе `Dockerfile`) и перезапускает контейнер.
- **Nginx (Reverse Proxy):** Установлен на ОС Ubuntu VPS. Слушает порты 80/443, привязывает SSL-сертификаты к доменам `coinlover.ru` / `coin.reloto.ru` и проксирует трафик в Docker-контейнеры на порты 8010/8011.

#### 📍 Расположение проектов на сервере
Все проекты лежат в домашней директории пользователя `root` на сервере `server.reloto.ru`:
- **Основное окружение (MAIN — coinlover.ru)**
  - Путь: `/root/MyProjects/coinLover`
  - Контейнер: `coinlover` (проброс порта 8010 -> 80)
  - Ветка: `main`
  - Секреты: `/root/MyProjects/coinLover/.env`
- **Тестовое окружение (DEV — coin.reloto.ru)**
  - Путь: `/root/MyProjects/coinlover-dev`
  - Контейнер: `dev-coinlover` (проброс порта 8011 -> 80)
  - Ветка: `preview`
  - Секреты: `/root/MyProjects/coinlover-dev/.env`

Конфигурации Nginx для доменов расположены по стандартному пути: `/etc/nginx/sites-available/`.

## 🔄 Sync Architecture Details
Проект использует схему **VPS Proxy** для работы с Google Sheets, реализованную на Node.js.

### 1. Почему это сделано:
- **CORS & Reliability:** Google Apps Script (GAS) блокирует запросы из браузера при ошибках доступа или редиректах. VPS Proxy позволяет отдавать четкие HTTP-статусы (например, 403 при истечении подписки) и управлять заголовками CORS.
- **Privacy:** Пользователям не нужно давать доступ к своим таблицам личному аккаунту разработчика, используется сервисный аккаунт.
- **Robustness:** Snapshot-based синхронизация предотвращает затирание данных при одновременной работе на двух устройствах или ручных правках таблицы.

### 2. Проверка доступа:
При каждом запросе бэкенд проверяет колонку `Access Ends` во вкладке `Users` (поддерживает форматы `DD.MM.YYYY` и `DD.MM.YY`). Если дата в прошлом, бэкенд возвращает 403 ошибку, и фронтенд (в `ModalManager`) показывает неудаляемую модалку блокировки с кнопкой связи в Telegram (`?text=...`).

## 📊 Data Parsing Rules
- **Flexible Headers:** Парсер в Node.js ищет ключи, игнорируя регистр и лишние пробелы. Поиск баланса поддерживает синонимы (balance, остаток, сумма).
- **Double Headers:** В листе `Transactions` используется две строки заголовков. Строка 1 — технические ключи (скрыты от пользователя), Строка 2 — человекочитаемые названия (зафиксированы для прокрутки).
- **Snapshot Sync:** Приложение запоминает состояние облака (снэпшот) и перед отправкой новых данных (Push) сверяет текущее состояние облака с этим снимком. Если есть отличия, вызывается модалка разрешения конфликтов. Это предотвращает "ложные конфликты" при локальных изменениях (например, после трансфера).

## 7. Решение технических проблем (Knowledge Base)

Во время разработки мультивалютности были решены критические баги, знание которых необходимо для поддержки проекта:

### 1. Конфликт Скролла и Drag-and-Drop
**Проблема:** Иконки кошельков «дергались» или не захватывались при включенном `scroll-snap-type`.
**Решение:** Удален `scroll-snap` из CSS для горизонтальных списков. Браузерные механизмы «прилипания» конфликтуют с абсолютным позиционированием `@dnd-kit`. Оставлен только `-webkit-overflow-scrolling: touch` для инерции.

### 2. Ошибки инициализации состояний (ReferenceError)
**Проблема:** Приложение падало с ошибкой `confirmDelete is not defined`.
**Решение:** Внедрен стандарт «All States at Top». Все хуки `useState`, `useRef` и `useFinance` должны быть объявлены в самом верху компонента `App.tsx` до начала объявления функций-обработчиков.

### 3. Липкий кэш при смене БД (Demo/Real)
**Проблема:** При переключении с Demo на Real данные не обновлялись.
**Решение:** Функция `toggleMode` теперь принудительно вызывает `localStorage.removeItem()` для всех ключевых данных перед перезагрузкой. Хук `useFinance` настроен на `pullSettings()`, если локальный кэш пуст.

### 4. Порядок столбцов в Google Sheets
**Проблема:** Добавление новых полей в середину таблицы ломало старые скрипты.
**Решение:** В `Code.gs` реализован поиск индексов по названиям заголовков (`col["Source Amount"]`). Теперь порядок столбцов в таблице может быть любым.

### 5. Визуальная асимметрия Drag-and-Drop (Трансфер vs Расход)
**Проблема:** Перемещение между кошельками (Трансфер) и перемещение из кошелька в категорию (Расход) выглядели по-разному, хотя механика одна и та же. Категории имели градиентный фон, а кошельки — стиль «монетки».
**Решение:** Внедрен единый стандарт `.draggable-coin` для всех интерактивных элементов (Кошельки, Категории, Доходы). 
* Унифицирован эффект `coin-target-glow` (масштабирование 1.1x, внешнее свечение и замена цвета иконки на белый).
* Синхронизированы состояния `isPressing`, `isSortingMode` и `grabbed-elevation` во всех компонентах.
* Теперь интерфейс ощущается целостным: любое перетаскивание «монетки» на «монетку» визуально предсказуемо.

### 6. Смещение монетки Drag-and-Drop на широких экранах
**Проблема:** При использовании приложения на мониторах с большой шириной (Desktop Chrome на Mac) перетаскиваемая «монетка» отображалась со значительным смещением вправо относительно курсора.

## 📄 Раздел: DND Manual (Механика Drag-and-Drop)

# Drag-and-Drop Mechanism (Hybrid Mobile-First Style)

Этот документ описывает логику взаимодействия с иконками в приложении CoinLover. 
Мы используем гибридный подход, чтобы обеспечить 100% стабильность на Android/iOS и Mac.

## 1. Режимы взаимодействия

| Действие | Условие срабатывания | Результат | Обработчик |
| :--- | :--- | :--- | :--- |
| **Клик (История)** | Короткое нажатие без движения | Открыть историю транзакций | Внешний `onClick` |
| **Редактирование** | Удержание иконки **2.0 сек** | Открывается модальное окно | `longPressTimerRef` |
| **Транзакция** | Захват иконки + движение вниз | Открывается Numpad | `dnd-kit` (Drag) |
| **Сортировка** | Удержание иконки **0.6 сек** | Включается режим перемещения | `sortingTimerRef` |

---

## 2. Техническая архитектура

### А. Разделение областей (The Icon-Only Grab Pattern)

Чтобы обеспечить свободный горизонтальный скролл и избежать случайных захватов, мы используем строгое разделение зон:

1. **Внешний контейнер (`setNodeRef`, `attributes`)**:
   - Отвечает за позиционирование элемента в сетке и анимации перемещения (`transform`).
   - Содержит стандартный React-обработчик `onClick`.
   - Клик срабатывает только если палец не сдвинулся и не включен режим сортировки.

2. **Внутренняя иконка-монетка (`listeners`)**:
   - **Только эта зона** является интерактивной для захвата (Drag-and-Drop).
   - Это позволяет пользователю прокручивать список (скроллить), касаясь области текста под иконкой или пространства между иконками.
   - Содержит `handlePointerDown`, который:
     - Запускает `listeners.onPointerDown(e)` для старта драга.
     - Активирует эффект «сжатия» (`scale-90`) для визуального отклика.
     - Запускает таймеры: **600мс** (сортировка) и **2000мс** (редактирование).

### Б. Визуальные состояния (Scale Logic)

Мы используем трехступенчатую систему масштабирования для интуитивного отклика:
1. **Покой**: `scale(1.0)`.
2. **Нажатие (Touch)**: `scale(0.9)` — монетка визуально «вжимается» в экран.
3. **Захват/Сортировка**: `scale(1.1)` — монетка «всплывает» над интерфейсом.

### В. Коллизии (Collision Detection)

В `App.tsx` используется кастомная стратегия:
- В обычном режиме коллизии ищутся только `pointerWithin` (палец над целью).
- В режиме сортировки используется `closestCenter`.
- Коллизия с самим собой (`active.id`) всегда игнорируется.

## 3. Анимация подсветки цели (Target Glow)

### Проблема
При перетаскивании кошелька над **категорией** подсветка мгновенно мигала и пропадала — в отличие от стабильной анимации при наведении на другой **кошелёк**.

### Решение

**1. `index.css`** — добавлен `transition` в `.coin-target-glow`:
```css
.coin-target-glow {
  background: var(--glass-border-highlight) !important;
  box-shadow: 0 0 20px var(--glass-border-highlight) !important;
  border-color: var(--text-muted) !important;
  transform: scale(1.10) !important;
  transition: transform 0.25s cubic-bezier(0.34, 1.56, 0.64, 1),
              box-shadow 0.25s ease,
              background 0.25s ease !important;
}
```

**2. Единый класс**: Все интерактивные drop-цели (Кошельки и Категории) теперь используют единый класс `.draggable-coin` и `.coin-target-glow` для полной визуальной симметрии трансферов и расходов.

## 4. Правила «Золотого кода»

Если нужно изменить поведение иконок:

1. **Никогда** не вешайте `listeners` на внешний контейнер. Это заблокирует скролл всего списка, так как любое касание будет перехвачено системой DnD.
2. **Всегда** ограничивайте `listeners` только круглой иконкой (`.draggable-coin`).
3. **Pointer-events**: Убедитесь, что у иконки **нет** класса `pointer-events-none`, иначе захват не сработает. Этот класс должен быть только у элементов внутри иконки (самой SVG или изображения).
4. **Тайминги**: Сортировка — **600мс**, Редактирование — **2000мс**. Порог движения (отмена клика) — **15px**.
5. **Сброс при Drag**: Обязательно добавляйте `useEffect`, который вызывает `clearTimers()`, если `isDragging` становится `true`.
6. **Единообразие**: Все элементы (Account, Category, Income) должны иметь одинаковую структуру и использовать класс `.draggable-coin` для базовых стилей.

## 5. Решение проблем с позиционированием (Wide Screens)

### Проблема (Coordinate Offset)
На экранах мониторов (Desktop Chrome на Mac) перетаскиваемая монетка отображалась со значительным смещением вправо относительно курсора. Это происходило из-за того, что `DragOverlay` находился внутри центрированного контейнера с `max-w-md mx-auto`, и система координат `dnd-kit` ошибочно учитывала боковые отступы (margin) родителя.

### Решение

1. **Глобальный контекст координат**:
   Обертка `DndContext` перенесена на уровень всего приложения (корневой `return`), а `DragOverlay` вынесен за пределы всех центрированных блоков. Теперь оверлей является прямым потомком `DndContext` и использует абсолютную систему координат окна браузера.

2. **Производительность и точность (No Transitions)**:
   Для классов `.draggable-coin` и `.grabbed-elevation` в `index.css` полностью удалены `transition: all` и `transition: transform`.
   - **Почему это важно**: CSS-анимация `transform` конфликтует с частотой обновления координат от JS-движка `dnd-kit`. Это вызывало «резиновый» эффект (задержку) и визуальный лаг при быстром движении мыши или пальца.
   - **Результат**: Монетка мгновенно следует за курсором без рывков и смещений.

## 📄 Раздел: Будущая архитектура: Шаги перехода на Multi-Tenant

# Шаги по переходу на Multi-Tenant архитектуру (CoinLover)

Переход приложения CoinLover к multi-tenant (многопользовательской) архитектуре означает, что один экземпляр приложения будет обслуживать множество независимых пользователей (тенантов), при этом их данные будут надежно изолированы друг от друга.

Учитывая текущий стек (React/TypeScript на клиенте + Google Sheets (Code.gs) как БД), вот подробный пошаговый план:

## 1. Внедрение аутентификации и авторизации (Auth)

Прежде чем разделять данные, системе необходимо уметь отличать одного пользователя от другого.

* **Выбор провайдера:** Firebase Auth, Supabase Auth, Clerk или Telegram Web App Auth (если приложение работает как Telegram Mini App).
* **Фронтенд:**
  * Создать экраны входа/регистрации.
  * Внедрить `AuthContext` для хранения глобального состояния текущего пользователя.
  * Настроить защищенные маршруты (Protected Routes), перенаправляющие неавторизованных пользователей на страницу входа.
* **Результат:** При входе приложение получает уникальный `user_id` (или JWT-токен).

## 2. Выбор стратегии хранения данных (База данных)

Google Sheets имеет лимиты на количество ячеек (10 млн) и при больших объемах начинает тормозить. Есть три пути реализации multi-tenancy:

### Вариант А: Изоляция на уровне строк (Row-Level / Single Sheet)

* **Суть:** Все пользователи хранятся в одной базе данных (одной Google Таблице). Во все листы (`Transactions`, `Metadata`, `Accounts`) добавляется колонка `user_id`.
* **Плюсы:** Легкий старт, один источник правды.
* **Минусы:** Таблица быстро заполнится. Риск утечки данных при ошибке в коде (кто-то получит чужие транзакции). Масштабируемость крайне слабая.

### Вариант Б: Изоляция на уровне таблиц (Database-per-tenant) — **Рекомендуется для архитектуры Google Sheets**

* **Суть:** У каждого пользователя — своя собственная Google Таблица.
* **Как это работает:**
    1. Пользователь логинится.
    2. Пользователь создает копию вашей эталонной таблицы и вставляет её URL (или ID) в настройки CoinLover.
    3. Приложение сохраняет этот `spreadsheetId` в профиле пользователя (в локальном хранилище или в легковесной БД авторизации).
    4. При всех API запросах фронтенд передает этот `spreadsheetId` в Apps Script.
* **Backend (`Code.gs`):** Модифицируется для работы с `SpreadsheetApp.openById(requestParams.spreadsheetId)` вместо жестко зашитого ID.
* **Плюсы:** Бесконечная масштабируемость, данные пользователей лежат на их личных Google Дисках (высокое доверие), никаких лимитов для вас как разработчика.
* **Минусы:** Пользователю требуется сделать несколько дополнительных шагов для старта. При изменении архитектуры (добавлении новых колонок) нужно обновлять таблицы всех пользователей.

### Вариант В: Миграция на полноценную СУБД (PostgreSQL / Supabase)

* **Суть:** Полный отказ от Google Sheets в пользу реляционной базы данных с поддержкой Row Level Security (RLS).
* **Плюсы:** Это индустриальный стандарт для SaaS. RLS гарантирует защиту данных на уровне ядра БД. Нативные связи, высокая скорость.
* **Минусы:** Требуется переписать весь бекенд, перенос существующей логики (бекапы, триггеры) на серверную часть.

## 3. Модификация Backend-логики (`Code.gs`)

Если вы оставляете Apps Script:

* **Безопасность:** Добавить проверку токена или передачу `spreadsheetId`.
* **Динамические таблицы:** Убрать `const SPREADSHEET_ID = '...';`. Вместо этого извлекать его из параметра `e.parameter.spreadsheetId` (для `doGet`) или из `JSON.parse(e.postData.contents).spreadsheetId` (для `doPost`).

## 4. Адаптация Frontend-стора (`useFinance.ts` и др.)

* **API Вызовы:** Во все функции отправки и получения данных (`fetchTransactions`, `addTransaction` и т.д.) добавить передачу `user_id` или `spreadsheetId`.
* **Очистка State:** Добавить функцию `logout`, которая полностью стирает стейт транзакций, кошельков и категорий, а также чистит `localStorage`/`IndexedDB` от чужих данных, чтобы другой пользователь не увидел их на этом же устройстве.
* **Изоляция кэша:** При сохранении локальных данных дописывать `user_id` в ключ, например: `localStorage.setItem('coinLover_data_${userId}', ...)`

## 5. Адаптация UI/UX

* Добавление страницы «Профиль / Настройки аккаунта», где указан текущий логин пользователя.
* Кнопка «Выйти» (Logout).
* Интерфейс «Подключения базы» (если выбран Вариант Б): красивый онбординг, показывающий пользователю, как скопировать таблицу и вставить ID.
* Обработка ошибок авторизации (401/403) с автоматическим редиректом на экран логина.

## 6. Задачи по Облачной Синхронизации (Cloud Sync)

* Сейчас у вас есть фоновые скрипты создания бэкапов (`ConfigsArch`). Если выбрать "Вариант Б", каждый пользователь будет иметь такой скрипт в своей таблице, и бэкапы будут делаться на его стороне.
* Если выбрать единую серверную БД, потребуется настроить систему ежедневных дампов (например, с помощью cron на сервере).

## Резюме: С чего начать?

1. Решить, остаетесь ли вы на Google Sheets. Если да, то **Вариант Б "Таблица для каждого"** — идеален (работает по принципу "Bring Your Own Database").
2. Создать ветку `feature/multi-tenant`.
3. Добавить экран входа и Auth (например, через Telegram логин, если это удобно).
4. Переписать `useFinance` для работы с динамическим ID таблицы.
