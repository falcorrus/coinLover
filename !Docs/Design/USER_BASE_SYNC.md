# Синхронизация и привязка пользовательских баз

Этот документ описывает механизмы, обеспечивающие надежную работу пользовательских баз данных CoinLover на различных устройствах, особенно на iOS.

## 1. Идентификатор таблицы (SSID)
Основным ключом к данным пользователя является ID его Google Таблицы. Приложение поддерживает три способа получения и хранения этого ID.

## 2. Форматы ссылок
Для обеспечения надежности PWA (Progressive Web App) на iOS внедрена маршрутизация через пути:
- **Основной формат:** `coinlover.ru/s/[SSID]`
- **Legacy формат:** `coinlover.ru/?ssId=[SSID]`

Использование пути (`/s/ID`) является приоритетным для iPhone, так как Safari сохраняет полный путь в манифесте приложения при нажатии «На экран "Домой"».

## 3. Механизм Persistence (Бетонная привязка)
Для решения проблемы изоляции PWA-контейнеров на iOS используется многоуровневая проверка:

1. **URL Path:** Если в пути есть `/s/ID`, этот ID становится активным.
2. **URL Search:** Проверка параметра `?ssId=`.
3. **LocalStorage:** Основное хранилище для работы в браузере.
4. **Cookies (Bridge):** При заходе по ссылке ID записывается в Cookie на **1 год**. 
   - *Зачем:* Safari и PWA-контейнер на iOS часто имеют раздельный LocalStorage, но разделяют Cookies. Это позволяет иконке на рабочем столе "увидеть" ID, по которому пользователь заходил в браузере.

## 4. Режим Demo
- Включается через `coinlover.ru/?demo=true`.
- При активации Demo-режима все пользовательские ID (в LocalStorage и Cookies) принудительно очищаются.
- На лендинге кнопка "Demo" работает "умно": если в памяти уже есть ID пользователя, она вернет его в его базу, а не в демо-режим.

## 5. Инициализация (Flow)
При запуске `App.tsx` хук `useUsers` выполняет поиск ID в указанном выше порядке приоритетов. Если ID найден в URL или Cookies, он принудительно синхронизируется с LocalStorage для обеспечения консистентности.
