# Архитектура интеграции Telegram-бота в CoinLover

В данном документе проанализированы варианты интеграции Telegram-бота (`@coinlover_bot`), который предоставляет персонализированные данные из индивидуальных Google Sheets пользователей на основе их идентификаторов из мастер-таблицы (или бэкенда Supabase).

---

## 1. Ключевой вызов: Связывание Telegram-аккаунта с пользователем приложения
Когда новый пользователь нажимает кнопку в веб/мобильном приложении и переходит в Telegram, бот видит только его Telegram `user_id` / `chat_id` и `username`. Он не знает, какому аккаунту в CoinLover соответствует этот пользователь.

### Решение: Telegram Deep Linking (`tg://` или `https://t.me/`)
Это бесшовный и стандартный паттерн для авторизации:
1. **Генерация токена:** При клике на кнопку «Подключить Telegram-бота» в интерфейсе приложения, бэкенд (Supabase/VPS API) генерирует временный уникальный токен (например, `tg_auth_a1b2c3d4`) и связывает его с текущим `user_id` (срок жизни ~5-10 минут).
2. **Переход в Telegram:** Приложение перенаправляет пользователя по ссылке:
   `https://t.me/coinlover_bot?start=tg_auth_a1b2c3d4`
3. **Активация бота:** В Telegram открывается чат с ботом и внизу появляется кнопка **«Запустить» (Start)**.
4. **Связывание:** При нажатии «Start» Telegram отправляет боту команду:
   `/start tg_auth_a1b2c3d4`
5. **Авторизация:** Бот считывает токен, отправляет его на бэкенд CoinLover:
   - Бэкенд сопоставляет токен с пользователем, сохраняет его `telegram_chat_id` и `username` в базу данных.
   - Возвращает боту подтверждение и `google_sheet_id` пользователя.
6. **Результат:** Бот приветствует пользователя по имени. Все последующие сообщения от этого Telegram `chat_id` автоматически маппятся на его личный `google_sheet_id` без необходимости повторной авторизации.

## 2. Варианты технической реализации бота

Ниже представлены 3 основных архитектурных сценария реализации бэкенда бота, отсортированные по сложности и гибкости.

### Вариант А: Использование n8n (Low-Code / No-Code) — *Рекомендуемый*
> [!NOTE]
> Поскольку в проекте CoinLover уже используется **n8n** для автоматизации процессов, этот подход является самым быстрым и простым.

*   **Как это работает:**
    1. В n8n создается триггер **Telegram Trigger** (слушает сообщения от бота).
    2. При получении команды `/start <token>` n8n делает запрос в Supabase/Мастер-таблицу, связывает аккаунты и возвращает приветствие.
    3. При отправке текстовых команд (например, «Баланс», «Добавить расход») n8n берет сохраненный `telegram_chat_id`, находит соответствующий `google_sheet_id` в Supabase/Мастер-таблице и использует встроенную ноду **Google Sheets** для чтения/записи данных.
    4. n8n форматирует ответ и отправляет его обратно в Telegram.
*   **Сложность:** Экстремально низкая (1 день).
*   **Плюсы:**
    *   Не нужно писать и деплоить серверный код бота.
    *   Нативная и стабильная интеграция n8n с Google Sheets и Supabase (авторизация через Oauth2 / Service Account).
    *   Легко изменять структуру меню, добавлять новые кнопки и логику прямо в визуальном редакторе.
*   **Минусы:**
    *   Менее гибкая кастомизация интерактивных элементов (например, сложных Telegram WebApps или динамических inline-меню на лету), хотя базовые кнопки и Markdown-разметка поддерживаются на 100%.

---

### Вариант Б: Выделенный микросервис на Node.js / Python (на VPS `server.reloto.ru`)
*   **Как это работает:**
    1. На VPS разворачивается легковесный сервис на Python (`aiogram`) или Node.js (`telegraf`).
    2. Бот общается с Telegram через Webhooks (на VPS уже настроен Nginx и SSL для маршрутизации на нужный порт).
    3. Бот делает запросы к существующему VPS Proxy `/api/sheets/` для работы с таблицами пользователей, либо напрямую использует Google Sheets API с сервисным аккаунтом `coinlover-service-acc@baonlineru.iam.gserviceaccount.com`.
*   **Сложность:** Средняя (2-3 дня).
*   **Плюсы:**
    *   Полный контроль над кодом, высокая производительность и мгновенный отклик.
    *   Возможность генерации сложных графиков/отчетов в виде картинок (используя `matplotlib` / `canvas`) и отправки их пользователю.
    *   Идеальная интеграция с Telegram API (поддержка inline-режима, кастомных клавиатур, WebApps).
*   **Минусы:**
    *   Необходимо поддерживать еще один работающий процесс на VPS (мониторинг через systemd, логирование, обработка падений).
    *   Написание бойлерплейт-кода для работы с API Telegram и базами данных.

---

### Вариант В: Google Apps Script Web App (Serverless)
*   **Как это работает:**
    1. Вебхук Telegram настраивается прямо на URL развернутого Web App в Google Apps Script (GAS) мастер-таблицы.
    2. При получении вебхука, GAS парсит сообщение, считывает `telegram_id`, находит в листе `Configs` строку пользователя, получает его `google_sheet_id`.
    3. Открывает таблицу пользователя через `SpreadsheetApp.openById(sheetId)` и производит чтение/запись.
    4. Отправляет HTTP-ответ в API Telegram с текстом сообщения.
*   **Сложность:** Средняя (2 дня).
*   **Плюсы:**
    *   Полностью serverless: не нужен VPS, базы данных и сторонние сервисы. Все крутится на мощностях Google абсолютно бесплатно.
    *   Прямой и мгновенный доступ к Google Sheets без необходимости настройки API-ключей и сервисных аккаунтов.
*   **Минусы:**
    *   Крайне неудобная отладка (отсутствие локальной среды разработки, сложные логи).
    *   Холодный старт GAS-скрипта может занимать от 1.5 до 3 секунд, из-за чего бот будет казаться "задумчивым".
    *   Жесткие лимиты Google на время выполнения скриптов (до 30 сек на запрос) и количество ежедневных триггеров.

## 3. Сравнительная таблица решений

| Критерий | Вариант А: n8n (Рекомендуемый) | Вариант Б: VPS Микросервис | Вариант В: GAS Serverless |
| :--- | :--- | :--- | :--- |
| **Сложность разработки** | 🟢 Низкая (No-code / Visual) | 🟡 Средняя (Код + Деплой) | 🟡 Средняя (GAS / JS) |
| **Скорость реализации** | 🟢 Очень быстро (1 день) | 🟡 Средне (2-3 дня) | 🟡 Средне (2 дня) |
| **Стоимость хостинга** | 🟢 Бесплатно (в рамках текущего n8n) | 🟢 Бесплатно (на текущем VPS) | 🟢 Бесплатно (Google) |
| **Скорость отклика (Latency)**| 🟢 Высокая (< 300мс) | 🟢 Высокая (< 100мс) | 🔴 Низкая (1.5 - 3 сек) |
| **Удобство поддержки** | 🟢 Легко (визуальный мониторинг) | 🟡 Требует контроля логов | 🔴 Сложно (неудобные логи GAS) |
| **Интеграция с Google Sheets**| 🟢 Нативная, очень простая | 🟡 Через API библиотеки | 🟢 Нативная, встроенная в GAS |

## 4. Итоговая рекомендация по реализации

1. **Этап 1 (Быстрый старт / MVP):** 
   Использовать **Вариант А (n8n)** с интеграцией через **Deep Linking**. 
   Это позволит за несколько часов настроить бота, проверить гипотезу, протестировать удобство для пользователей и получить полностью рабочую связку Telegram <-> Приложение <-> Индивидуальные Google Sheets.
   
2. **Этап 2 (Масштабирование / Сложные фичи):**
   Если пользователям потребуются сложные интерактивные отчеты, красивые графики или полноценный Mini App (Web App) прямо внутри Telegram — перенести логику на **Вариант Б (микросервис на VPS)**, переиспользовав готовую структуру БД и схему связывания аккаунтов через Deep Linking.
