# Тезисы статьи: "Используем Google Sheets как базу данных: от наколеночного MVP до безопасного мобильного приложения с Passkeys"

Данный план статьи детально описывает практический опыт разработки реального PWA/мобильного приложения (на примере CoinLover) с использованием Google Таблиц в роли хранилища данных. Статья ориентирована на разработчиков, которые хотят создавать проекты быстро, дёшево и с минимальным бэкендом, не жертвуя при этом безопасностью и удобством пользователей.

---

## Введение: Зачем вообще использовать Google Sheets в 2026 году?

* **Плюсы решения:**
    * **Бесплатность и бесконечный лимит:** Никаких счетов за СУБД на раннем этапе.
    * **Идеальный "админ-интерфейс" по умолчанию:** Пользователь видит свои данные в виде привычной таблицы, может сам строить графики, писать формулы и выгружать отчеты.
    * **Абсолютная прозрачность:** Данные принадлежат пользователю, они лежат на его Google Диске (или в таблице, которой он управляет), что повышает доверие.
* **Минусы и вызовы:**
    * **Отсутствие индексов и медленный отклик:** Запросы к Sheets API могут занимать 0.5–2 секунды.
    * **Проблема CORS и безопасности:** Нельзя делать запросы к Google API напрямую из браузера/мобильного приложения без раскрытия ключей.
    * **Concurrency (одновременный доступ):** Sheets API не предназначен для высоконагруженных систем с миллионами транзакций в секунду.
    * **Лимиты API:** Ограничения на количество запросов в минуту (Read/Write quotas).

---

## Раздел 1. Архитектура решения: Правильный гибридный стек

* **Клиентский слой:** React 19 + TypeScript + Vite.
    * Использование Capacitor (Capacitor 8) для сборки нативных мобильных приложений (Android/iOS).
    * **Решение проблемы CORS:** Использование `universalFetch` и `CapacitorHttp` на мобильных устройствах для выполнения запросов в обход ограничений браузера.
* **Серверный мост (VPS Proxy):** Node.js/Express сервер на VPS (`server.reloto.ru`).
    * **Почему это критично:** Никогда не храните ключи доступа к Google API в клиентском коде.
    * Бэкенд-прокси решает задачи безопасности, кэширования, проверки подписок и логирования транзакций.
* **Схема авторизации через Service Account:**
    * Вместо сложного OAuth-процесса для каждого пользователя (требующего согласия Google и верификации приложения), используется единый сервисный аккаунт (`coinlover-service-acc@...`).
    * Пользователь просто расшаривает свою таблицу на этот email с правами редактора. Всё! Вход бесшовный, настройка занимает 5 секунд.

## Раздел 2. Безопасность без компромиссов: Беспарольный вход и Passkeys

* **Проблема безопасности "голых" ID таблиц:**
    * ID Google Таблицы (44 символа) — это криптографически стойкая случайная строка. Её невозможно угадать перебором. Но если злоумышленник перехватит этот ID, он сможет читать данные через прокси.
* **Внедрение бесшовной регистрации по Passkey (WebAuthn / Face ID / Fingerprint):**
    * **Шаг 1:** На первом шаге онбординга пользователь создает Passkey, который связывается с его контактом (email или Telegram).
    * **Шаг 2:** После подключения таблицы открытые ключи биометрии (Public Keys) сохраняются в конфигурационной вкладке самой таблицы пользователя.
    * **Шаг 3 (Вход в один клик):**
        * Пользователь инициирует вход.
        * Сервер ищет контакт в таблице пользователей (Master Sheet).
        * Происходит нативная аутентификация по биометрии на устройстве (`@simplewebauthn/browser`).
        * Сервер верифицирует подпись, определяет приватный `ssId` пользователя и отдаёт сессию.
    * Пароли отсутствуют как класс. Полная защита от фишинга и утечек.
* **Нативная интеграция Passkey на Android (Capacitor):**
    * Реализация `NativeAuthScreen` с декодированием контакта и поиском ID таблицы.
    * Решение проблемы с таймаутами Google Play Services на Android Chromium через префетч (Prefetch) опций регистрации WebAuthn для сохранения User Gesture.
* **Защита Мастер-таблицы (Master Sheet):**
    * Полное удаление `MASTER_SS_ID` из клиентского кода.
    * Проверка прав доступа, валидация подписок (`Access Ends` колонка) и админские операции выполняются исключительно на стороне сервера. Блокирующий экран (lock modal) в UI при истечении доступа.

## Раздел 3. Синхронизация данных и борьба с ограничениями

* **Конфликты версий (Conflict Resolution):**
    * Как не затереть новые данные в облаке старым локальным состоянием?
    * Внедрение схемы сравнения таймстампов (Timestamps) локальной сессии и облачной таблицы.
    * При обнаружении конфликта — показ диалогового окна: "Обновить локальные данные" или "Перезаписать облако".
* **Робастный парсер неструктурированного ввода:**
    * Пользователи могут случайно переименовать колонки в своей таблице.
    * **Решение:** Гибкий парсер заголовков, игнорирующий регистр символов и пробелы. Использование синонимов (например, `"Balance" = "Остаток" = "Сумма"`).
* **Специфика мультивалютности:**
    * Хранение транзакций с полями `Source Amount` (счет списания) и `Target Amount` (целевой счет) + курсы конвертации.
    * Ограничение визуальных валютных символов (только популярные: $, €, £) для чистоты UI и перенос остальных на трехбуквенные коды (RUB, RSD) во избежание поломок верстки.

## Раздел 4. Оптимизация производительности (Как сделать интерфейс "летающим")

* **Борьба с сетевыми задержками (Latency):**
    * **Оптимистичный UI (Optimistic Updates):** Изменения отображаются на экране мгновенно, а реальный запрос к Sheets API уходит в фоне.
    * **Локальное кэширование:** Использование LocalStorage / IndexedDB как основного источника данных при запуске приложения, синхронизация с облаком происходит асинхронно.
* **Предотвращение мерцания интерфейса (Layout Thrashing & FOUC):**
    * Интеграция темной темы (Dark Glassmorphism Premium) без эффекта FOUC (Flash of Unstyled Content) на этапе загрузки страницы.
    * Архитектурное разделение тяжелых компонентов (например, фиксированный плеер для сторис работает параллельным оверлеем, не перегружая DOM-дерево основного экрана).

## Заключение: Вердикт и область применения

* Google Sheets — это **не** серебряная пуля, но это мощнейший инструмент для инди-хакеров, стартапов на стадии валидации гипотез (MVP) и личных утилит.
* При правильном подходе (VPS Proxy + Passkeys + Optimistic UI) пользователь получает премиальный опыт работы с мобильным приложением, даже не подозревая, что под капотом крутится обычная Google Таблица.
* Архитектура готова к плавному переезду на PostgreSQL / Supabase, когда лимиты Google Sheets станут тесными — достаточно будет переписать методы бэкенд-прокси, сохранив фронтенд без изменений.
