# Аудит безопасности интеграции Google Sheets в CoinLover

В данном отчете анализируется безопасность хранения и передачи ссылок (ID) Google Таблиц в клиентском приложении (PWA, Capacitor Android/iOS) и на бэкенде (VPS Proxy).

---

## 1. Резюме (Executive Summary)

* **Можно ли извлечь секретные ключи доступа к Google API из приложения?**  
  **НЕТ.** Секретный ключ сервисного аккаунта (Google Service Account JSON) и учетные данные хранятся исключительно на сервере (VPS) в файле `google-credentials.json` или в переменных окружения бэкенда. В скомпилированном коде приложения (клиентском бандле) секретных ключей нет.

* **Можно ли извлечь ID таблиц (ссылки) из приложения?**  
  **ДА.** 
  1. **ID Мастер-таблицы** (`1IQCs35RQlMMQsGB-CRczJeuRqa8WIxW4Sy_kjZyHP2M`) зашит в клиентском коде (`src/App.tsx`) и закомментирован в `capacitor.config.ts`. Злоумышленник может легко извлечь его путем декомпиляции APK/IPA или через DevTools в браузере.
  2. **ID личных таблиц пользователей** хранятся локально на их устройствах (в LocalStorage) и передаются в параметрах запросов к бэкенду. Они могут быть перехвачены только при физическом доступе к устройству или в случае утечки базы данных/логов сервера.

---

## 2. Архитектура безопасности (Текущее состояние)

В CoinLover реализована правильная архитектурная схема проксирования запросов:
1. Клиентское приложение **не общается напрямую с Google Sheets API**.
2. Все запросы идут через бэкенд-прокси на сервере: `/api/sheets` (домены `coinlover.ru` и `coin.reloto.ru`).
3. Бэкенд авторизуется в Google API с помощью **Service Account** (`coinlover-service-acc@baonlineru.iam.gserviceaccount.com`), который имеет права редактора на таблицы.
4. Клиент передает только `ssId` (ID таблицы) в качестве параметра, а бэкенд выполняет операции чтения/записи от лица сервисного аккаунта.

## 3. Анализ угроз и уязвимостей (Threat Analysis)

Несмотря на безопасное хранение ключей, в текущей реализации бэкенд-прокси и клиента есть несколько слабых мест:

### Угроза А: Утечка ID Мастер-таблицы и несанкционированный доступ
* **Механизм уязвимости:** В коде бэкенда `api/sheets.ts` проверка подписки пользователя (активности доступа) отключается, если запрашиваемый ID совпадает с мастер-таблицей:
  ```typescript
  if (ssId && ssId !== MASTER_SS_ID) { ... // проверка }
  ```
* **Последствия:** Злоумышленник, зная `MASTER_SS_ID` (который зашит на фронтенде), может отправить GET/POST запросы к `/api/sheets?ssId=1IQCs35RQlMMQsGB-CRczJeuRqa8WIxW4Sy_kjZyHP2M` и без какой-либо авторизации прочитать или записать транзакции в конфигурационную и транзакционную вкладки мастер-таблицы администратора.
* **Смягчающий фактор:** Вкладка `Users` (список всех пользователей, их email и ID их таблиц) бэкендом в ответах клиенту **не возвращается** (она читается только внутри бэкенда для проверки прав). Финансовые транзакции администратора (Евгения), если они ведутся в мастер-таблице, при этом полностью открыты для чтения тем, кто отправит запрос с этим ID.

### Угроза Б: Отсутствие авторизации запросов к личным таблицам
* **Механизм уязвимости:** Бэкенд CoinLover не проверяет сессию пользователя (JWT / токен авторизации) при запросах к конкретной таблице. Он доверяет параметру `ssId`.
* **Последствия:** Если злоумышленник каким-либо образом узнает ID таблицы другого пользователя (например, пользователь случайно поделится ссылкой), злоумышленник сможет читать и записывать туда транзакции через ваш бэкенд-прокси.
* **Ослабленная валидация (Allowing by default):** Если ID таблицы вообще нет в списке `Users` в мастер-таблице, бэкенд разрешает доступ по умолчанию (`Allowing by default`), чтобы не блокировать новых пользователей и демо-режим. Это означает, что любая расшаренная на сервисный аккаунт таблица доступна любому, кто знает её ID.
* **Смягчающий фактор:** ID Google Таблицы — это криптографически стойкая случайная строка длиной 44 символа (например, `1IQCs35RQlMMQsGB-CRczJeuRqa8WIxW4Sy...`). Угадать её перебором невозможно. Безопасность данных пользователей сейчас держится на строгой конфиденциальности этой ссылки.

## 4. Рекомендации по устранению рисков

Для повышения уровня безопасности до премиального (Senior Engineer Level) рекомендуется провести следующие доработки:

### 1. Защита Мастер-таблицы (Высокий приоритет)
* **Действие:** Полностью исключить `MASTER_SS_ID` из клиентского кода.
* **Реализация:**
  - На фронтенде в `src/App.tsx` заменить хардкод `MASTER_SS_ID` на динамический запрос шаблона (`/api/sheets?action=template`), который возвращает дефолтную структуру без раскрытия ID.
  - На бэкенде в `api/sheets.ts` ограничить выполнение действий с `MASTER_SS_ID` (например, разрешить их только при наличии секретного токена администратора `X-Admin-Token` в заголовках запроса).

### 2. Привязка таблиц к учетным записям (Средний приоритет)
* **Действие:** Защитить запросы к `/api/sheets` авторизацией.
* **Реализация:**
  - Привязать каждый `ssId` к авторизованному Email/Telegram пользователя (которые уже хранятся в вашей базе `Users`).
  - При отправке запроса с клиента передавать токен авторизации (например, Supabase JWT или верифицированный email), а на бэкенде строго сверять: *«Имеет ли право текущий авторизованный пользователь делать запросы к таблице с данным ssId?»*.
