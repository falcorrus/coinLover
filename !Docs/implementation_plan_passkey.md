# Implementation Plan: Интеграция Passkey в CoinLover с децентрализованным хранением в Google Таблицах

Этот план описывает добавление альтернативного, полностью безопасного метода входа через **Passkey (Face ID / Touch ID / Windows Hello)**. 

Новый функционал будет полностью опциональным, существующий вход по прямой ссылке (`ssId`) и QR-коду сохранится без изменений. Публичные ключи пользователей будут храниться непосредственно в их персональных Google Таблицах, что исключает необходимость ведения базы данных пользователей на нашем VPS-сервере.

---

## 📌 Архитектура решения

```
┌─────────────────┐             ┌─────────────────────────┐             ┌─────────────────────┐
│  React Frontend │             │     Node.js VPS Proxy   │             │   Google Sheets     │
│                 │             │                         │             │                     │
│  1. Регистрация ├────────────>│  1. Валидация подписи   ├────────────>│  Сохранение ключа   │
│  2. Вход        │<────────────┤  2. Проверка challenge  │<────────────┤  в лист Configs     │
└─────────────────┘             └─────────────────────────┘             └─────────────────────┘
```

### 1. Как это работает:
* **Регистрация ключа (после входа):** Пользователь заходит в приложение стандартным образом. В настройках профиля появляется плашка «Включить вход по Face ID / Passkey». При включении генерируется Passkey. Устройство запоминает домен `coinlover.ru` и сопоставляет его с `ssId` таблицы (записывается в `user.id` как Resident Key). Публичный ключ отправляется на бэкенд и сохраняется в его личной таблице (лист `Configs`).
* **Вход на новом устройстве:** На экране лендинга появляется кнопка «Войти через Face ID / Passkey». Устройство находит Passkey для домена, запрашивает биометрию, подписывает случайный вызов (challenge) и возвращает фронтенду цифровую подпись и зашифрованный `ssId`. Бэкенд запрашивает публичный ключ из таблицы пользователя по `ssId`, проверяет подпись, и при успехе авторизует сессию.

### 🧠 Компонент 3: Backend (Node.js VPS Proxy)

#### [MODIFY] [api/sheets.ts](file:///Users/eugene/MyProjects/CoinLover/api/sheets.ts) (бизнес-логика бэкенда)
Необходимо внедрить 4 новых роута для WebAuthn-авторизации:

1. **`GET /api/auth/register-options`**:
   * Генерирует параметры для создания ключа с помощью `generateRegistrationOptions()` из `@simplewebauthn/server`.
   * Сохраняет сгенерированный `challenge` во временную сессию (или JWT-cookie) на клиенте для последующей сверки.
2. **`POST /api/auth/register-verify`**:
   * Принимает подпись регистрации устройства от фронтенда.
   * Вызывает `verifyRegistrationResponse()`.
   * После успешной проверки сохраняет полученный `publicKey` (в формате Base64/PEM) и `credentialId` в таблицу пользователя (лист `Configs`, новые системные ключи настроек).
3. **`GET /api/auth/login-options`**:
   * Генерирует параметры авторизации с помощью `generateAuthenticationOptions()`.
   * Сохраняет `challenge` во временную сессию.
4. **`POST /api/auth/login-verify`**:
   * Принимает подпись вызова, `credentialId` и `ssId` таблицы.
   * С помощью сервисного аккаунта запрашивает лист `Configs` указанной таблицы, вытягивает `Passkey_Public_Key` and `Passkey_Credential_ID`.
   * Вызывает `verifyAuthenticationResponse()` для криптографической сверки подписи с полученным публичным ключом.
   * Если проверка пройдена — возвращает клиенту успешный статус и подтверждает сессию.

## ❓ Открытые вопросы (Требуется обратная связь)

> [!IMPORTANT]
> **1. Метод генерации Challenge (Вызова):**
> Для проверки подписи бэкенд должен сравнить присланный challenge с тем, который он выдал. Так как у нас нет постоянной базы данных сессий на бэкенде, самый чистый способ передавать challenge — сохранять его в зашифрованную httpOnly cookie (`session_challenge`) с коротким временем жизни (например, 2 минуты). Вы согласны с таким подходом?
>
> **2. Размещение ключей в Google Таблице:**
> Мы можем хранить ключи авторизации прямо в листе `Configs` (в виде пар ключ-значение, как и остальные настройки). Это самый простой путь. Альтернатива — автоматически создавать отдельный скрытый лист `Security`. Хранение в `Configs` является более простым и устойчивым к изменениям. Вы предпочитаете `Configs`?
>
> **3. Поддержка Capacitor (Мобильные устройства):**
> Планируется ли в ближайшее время сборка мобильного приложения под iOS/Android? Если да, то для бесшовной работы Passkeys потребуется сгенерировать файл `apple-app-site-association` и `.well-known/assetlinks.json` на сервере для поддержки Associated Domains. Если пока фокус только на Web/PWA, этот шаг можно отложить.
